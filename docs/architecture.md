# Архитектура

> Свой сервер World of Warcraft на **.NET 10** + **MySQL**.
> Цель: **играбельный сервер** под **WotLK 3.3.5a (build 12340)**.
> Эталон для портирования: **CMaNGOS-WotLK** (`cmangos/mangos-wotlk`), дополнительно TrinityCore 3.3.5 и WCell (C#).

Дорожная карта и статусы вех — в [roadmap.md](roadmap.md).

## 0. Принципы написания кода

Код пишем согласно принципам **SOLID**:

- **S — Single Responsibility.** Один класс — одна ответственность. Не плодим «god-классы»
  (репозиторий на все таблицы, хендлер на всё сразу). Пример: DAL разбит на focused-репозитории
  (`EfCharacterRepository`/`EfInventoryRepository`/`EfQuestRepository`/`EfCharacterStateRepository`),
  а не один `EfCharacterStore` на всё.
- **O — Open/Closed.** Расширяем поведение новыми типами/обработчиками, а не правкой существующих
  (реестр опкодов: новый опкод = новый метод с атрибутом, без центрального switch).
- **L — Liskov.** Реализации интерфейсов взаимозаменяемы без сюрпризов для вызывающего.
- **I — Interface Segregation.** Узкие интерфейсы; потребитель зависит только от того, что реально
  использует (сегрегированные интерфейсы DAL вместо одного жирного).
- **D — Dependency Inversion.** Зависим от абстракций (интерфейсов DAL), а не от конкретных классов;
  внедрение через DI.

При появлении «жирного» класса с несколькими ответственностями — разбиваем на SRP-классы.

## 1. Что такое эмулятор сервера WoW

Эмулятор — это набор сетевых служб, которые «притворяются» официальным сервером для немодифицированного клиента `Wow.exe` (3.3.5a, build 12340). Клиент сам по себе содержит всю графику/звук/карты; сервер отвечает за аутентификацию, состояние мира и игровую логику.

```
┌──────────────┐   1. SRP6 login    ┌───────────────┐
│   Wow.exe    │ ◄────────────────► │  AuthServer   │  логин + список реалмов
│  build 12340 │      (TCP 3724)    │  (realmd)     │
│              │                    ├───────────────┤
│              │   2. world proto   │  WorldServer  │  вся игровая логика
│              │ ◄────────────────► │  (mangosd)    │  (TCP 8085)
└──────────────┘   (RC4 encrypted)  └───────┬───────┘
                                            │
                          ┌─────────────────┼─────────────────┐
                      MySQL             DBC / maps         (вынесено из
                  auth/chars/world   (данные клиента)        клиента)
```

## 3. Сетевой протокол 3.3.5a (ключевые узлы)

### 3.1 Аутентификация — SRP6
- Логин-сервер использует **SRP6** (Secure Remote Password): обмен `B`, `N`, `g`, `s`; клиент доказывает знание пароля без его передачи.
- Результат — **session key (K)**, который потом используется для шифрования world-трафика.
- Логины/пароли хранятся как SRP-верификатор `v` и соль `s` (не как хэш пароля).
- Реализация: `AlexWoW.Cryptography` (`Srp6`, `Srp6Server`, `Srp6Client`).

### 3.2 Шифрование world-пакетов — RC4 (важное отличие от ванили!)
- В **3.3.5 заголовки пакетов шифруются RC4**, ключи выводятся через **HMAC-SHA1** из session key.
  - Серверный ключ (encrypt): HMAC-SHA1(K) с серверным seed.
  - Клиентский ключ (decrypt): HMAC-SHA1(K) с клиентским seed.
  - Делается «прогрев» (drop) первых ~1024 байт RC4-потока.
- Шифруется только **заголовок** (размер + opcode), тело пакета — открытое.
- Без корректной реализации этого шага клиент не войдёт в мир (разорвёт соединение).

### 3.3 Opcodes
- Сотни опкодов: `CMSG_*` (от клиента), `SMSG_*` (к клиенту), `MSG_*` (двусторонние).
- Архитектура обработки: **реестр opcode → handler** (реализовано). Методы помечены
  `[WorldOpcodeHandler(...)]`, `WorldPacketRouter` строит таблицу через рефлексию (стиль WCell).
  Обработчики сгруппированы по фичам в `Handlers/`; `WorldSession` — только транспорт + состояние.
- Старт-набор для «входа в мир»: `CMSG_AUTH_SESSION`, `SMSG_AUTH_RESPONSE`, `CMSG_CHAR_ENUM`/`SMSG_CHAR_ENUM`, `CMSG_PLAYER_LOGIN`, `SMSG_LOGIN_VERIFY_WORLD`, набор начальных `SMSG_*` (account data, tutorial, time).

### 3.4 Сериализация
- Бинарный little-endian; есть **bit-packed** структуры (особенно movement в 3.3.5).
- **ObjectUpdate (SMSG_UPDATE_OBJECT)** — самый сложный пакет: update-mask поля объектов, движение, создание/обновление сущностей. Это сердце «видимости» мира.

## 4. Данные

### 4.1 Клиентские данные (извлекаются из клиента 3.3.5a)
- **DBC** (`DBFilesClient/*.dbc`) — спеллы, предметы, карты, расы/классы и т.д. Нужно парсить.
- **maps / vmaps / mmaps** — геометрия для коллизий и навигации (pathfinding). Извлекаются утилитами-экстракторами из MPQ клиента.
- Эти данные **не распространяем** — пользователь генерирует их из своей копии клиента.

### 4.2 Серверная БД мира
- В CMaNGOS-WotLK мир поставляется **MySQL-дампом** (спавны, лут, квесты, тексты, скрипты) — грузим почти как есть, **конвертация не нужна** (ключевая причина выбора MySQL).
- Две физические БД: **`alexwow_auth`** — НАШИ изменяемые данные (аккаунты `account` — SRP6 соль/верификатор + `email` для веб-панели, M8; реалмы, персонажи `characters`/`character_*`, `account_data`); **`mangos`** — read-only дамп мира (статика: спавны, шаблоны, квесты, спеллы `spell_template`, …).

### 4.3 Слой доступа к данным (DAL) — `AlexWoW.Database`
Гибрид (рефактор #23–25, см. [devlog/M7-tech-debt.md](devlog/M7-tech-debt.md)):
- **EF Core 9 + Pomelo (MySQL)** для `alexwow_auth`: сущности (`Entities/`) + `AuthDbContext` + **миграции** (вместо ручных `CREATE/ALTER`). На прод-БД миграции адаптированы baseline-строкой в `__EFMigrationsHistory` (таблицы не пересоздаются). ⚠️ Под EF Core 10 релиза Pomelo ещё нет — EF9 работает на рантайме net10 через roll-forward.
- **Dapper** для read-only дампа `mangos` (таблицы по 100+ столбцов, тяжёлые джойны — миграции не нужны).
- Потребители (хендлеры, сессии, `*Store`) зависят от **интерфейсов-репозиториев** (`Abstractions/`), не от конкретных классов и не от голого SQL: `IAccountRepository`/`IRealmRepository`/`ISchemaInitializer`, `ICharacterRepository`/`IInventoryRepository`/`IQuestRepository`/`ICharacterStateRepository`, и focused-репозитории mangos (`ICreatureRepository`/`IItemTemplateRepository`/`IVendorRepository`/`ITrainerRepository`/`ILootRepository`/`IQuestTemplateRepository`/`IFactionRepository`/`IPlayerDataRepository`/`IGameObjectRepository`) за композитным фасадом `IWorldRepository`.

## 5. Технологический стек (.NET)

| Слой | Выбор | Обоснование |
|---|---|---|
| Рантайм | **.NET 10** | Современный async, высокая производительность |

## 7. Риски и заметки

- **Объём.** Полноценный играбельный сервер — это годы. Поэтому строго идём по вехам, каждая даёт проверяемый результат.
- **БД — MySQL.** Совпадает с форматом дампов эталонов → конвертация не нужна. Минус: MySQL-специфика (типы, движок InnoDB) — закладываемся на это в схеме.
- **Bit-packed movement в 3.3.5.** Один из самых нетривиальных участков протокола — закладываем время.
- **RC4 header crypt.** Критичен для M2; реализуем и тестируем отдельно по эталону.
- **Данные клиента.** DBC/maps/vmaps/mmaps генерируются пользователем из своей копии клиента; в репозиторий не кладём.
- **Правовой статус.** Эмуляция в образовательных целях; данные Blizzard не распространяем.

## 8. Клиентские аддоны (dev-тулинг)

Клиентские аддоны WoW (Lua + FrameXML, интерфейс 3.3.5a = `30300`) — вспомогательный слой поверх обычного
протокола: они **не имеют привилегий** и общаются с сервером теми же пакетами, что и UI клиента. Исходники
ведём в репозитории (`tools/addons/`), рабочая копия ставится в `Interface\AddOns\` клиента (вне репо).

### AlexDevCmd — пульт dev-команд
Древовидная панель для расстановки dev-сущностей и прочих dev-команд (`.trainer`, `.craft`, `.level`,
`.additem` и т.п.) без ручного ввода в чат.

- **Транспорт.** Аддон отправляет команду как обычный **SAY**-чат (`SendChatMessage(".trainer mage", "SAY")`).
  Сервер ловит сообщения, начинающиеся с `.` (`WorldServer/Handlers/DevCommands.TryHandleAsync`), выполняет
  команду и **не эхоит** её в чат. Никакого спец-протокола/опкода не вводили.
- **Безопасность.** Гейт **`is_admin` остаётся серверным** — аддон ничего не обходит; под обычным аккаунтом
  сервер отклонит команду. То есть аддон — лишь удобный «генератор» тех же чат-команд.
- **UI.** Своя vanilla-панель (`FauxScrollFrame`, сворачиваемые категории), без внешних библиотек. Каталог
  команд **захардкожен** в Lua (таблица `TREE`) — при добавлении новых dev-команд на сервере его дополняют
  вручную (авто-синк каталога через addon-message — возможное развитие, пока не нужен). Свободные значения
  (`.level`/`.xp`/`.additem`/`.learn`/`.buff`) запрашиваются окном `StaticPopup` с полем ввода.
- **Запуск.** Слэш `/dev` (или `/devcmd`); позиция окна — в SavedVariable `AlexDevCmdDB`.

**Принцип:** dev-инструменты для клиента строим как аддоны поверх существующих команд/пакетов, не давая им
привилегий и не добавляя серверных «бэкдоров»; авторитет и гейтинг — на сервере.

## 9. Веб-панель игрока (`AlexWoW.Web`, M8)

Сайт для игроков: регистрация, вход, смена пароля, просмотр персонажей. **ASP.NET Core Razor Pages
(.NET 10)**, за **Caddy** на homeserver, домен **`alexwow.home.srv`**. Ход работ и решения —
[devlog/M8-web-interface.md](devlog/M8-web-interface.md).

- **Та же БД, тот же SRP6.** Панель работает с `alexwow_auth` через те же focused-репозитории EF
  (`IAccountRepository`/`ICharacterRepository`/`IInventoryRepository`), а соль+верификатор считает
  теми же вызовами `Srp6.CalculateVerifier`, что и CLI `create-account`. Пароль не хранится; аккаунт
  с сайта работает в игре, смена пароля сразу действует в клиенте. Миграции применяет **AuthServer**
  (единый мигратор) — панель только читает/пишет.
- **⚠️ Email ≠ игровой логин.** Клиент WoW 3.3.5a режет поле «Учётная запись» по «@» (это
  разделитель `account@логин-сервер`), поэтому **email как игровой логин даёт дисконнект**. Решение:
  email — для входа на сайт (`account.email`, nullable+unique), а в игру вход по отдельному **имени
  аккаунта** (латиница/цифры); **SRP6-верификатор считается по игровому имени**.
- **Безопасность.** Cookie-аутентификация; страницы под `AuthorizeFolder("/")` (кроме
  `Index`/`Login`/`Register`); детали персонажа доступны только владельцу (проверка `AccountId`);
  ключи Data Protection — в named-volume (переживают пересборку контейнера).
- **Развёртывание.** Runtime-only образ (`Dockerfile.web`, `aspnet:10.0`), сервис `alexwow-web`
  (порт 8090→8080), Caddy проксирует `https://alexwow.home.srv` → `192.168.2.210:8090`. Игровой
  протокол (3724/8085) — сырой TCP, **за Caddy не идёт**. Подробности — [deployment.md](deployment.md).
