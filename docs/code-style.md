# Code-style AlexWoW

Обязателен к прочтению перед правками кода. База — [конвенции Microsoft C#](https://learn.microsoft.com/dotnet/csharp/fundamentals/coding-style/coding-conventions); ниже зафиксированы наши уточнения и отступления. Машинная часть — в `.editorconfig` (корень), общие свойства сборки — в `Directory.Build.props` (`net10.0`, `Nullable`, `ImplicitUsings`, `LangVersion=latest`, `TreatWarningsAsErrors`).

## 1. Язык

- XML-доки и комментарии — **на русском**. Термины протокола/игры (опкоды, spell, aura, GUID) не переводятся насильно.
- Теги вех и задач в комментариях (`M6.4`, `M7 #33`, `Devcommands #79`) — сохраняются и добавляются: это связь кода с историей решений.
- Комментарий объясняет **почему** (ограничение протокола, поведение клиента, сверка с эталоном), а не что делает строка.
- Логи Serilog — сообщения на русском, структурные параметры в шаблоне: `"TP '{User}' (та же карта {Map})"`.

## 2. Раскладка проектов

| Проект | Ответственность |
|---|---|
| `AlexWoW.Common` | Бинарные читалка/писалка пакетов (ByteReader/ByteWriter), без зависимостей |
| `AlexWoW.Cryptography` | SRP6, RC4, дайджесты — чистая криптография, без I/O |
| `AlexWoW.Database` | DAL: EF Core (изменяемая `alexwow_auth`) + Dapper (read-only дамп `mangos`); интерфейсы в `Abstractions/`, реализации в `Repositories/` |
| `AlexWoW.DataStores` | Файловые данные клиента: `.map` (террейн), `.vmap` (коллизии/LoS), `.mmap` (навмеш) |
| `AlexWoW.AuthServer` | Логин-сервер (SRP6, реалм-лист), порт 3724 |
| `AlexWoW.WorldServer` | Мир: `Net/` (транспорт, сессия), `Handlers/` (опкод-модули и игровые сервисы), `World/` (состояние мира, сторы), `Protocol/` (билдеры пакетов, константы) |
| `AlexWoW.Web` | Панель игрока (Razor Pages) |

## 3. Нейминг и форматирование

- PascalCase: типы, методы, свойства, константы (включая приватные `const` и `static readonly`).
- `_camelCase`: приватные инстанс-поля. camelCase: параметры, локальные.
- Интерфейсы — `I…`; асинхронные методы — суффикс `Async`; `CancellationToken` — последним параметром (имя `ct`).
- File-scoped namespaces, `var` везде, скобки Allman, отступ 4 пробела, строка ≤ 120 символов.
- Один публичный тип на файл. Мелкие связанные типы (enum результата, record-DTO) допустимы рядом с владельцем.
- Иммутабельные модели — `record` / `readonly record struct`; мутабельные — `class`.

## 4. Политика статических классов

Статический класс допустим **только для хелперов** — чистых stateless-функций без I/O и без зависимостей на сервисы:

**Разрешено (whitelist):**
- Билдеры пакетов `WorldServer\Protocol\*` (PlayerSpawn, QuestPackets, SpellPackets, MovementPackets, …) — детерминированные функции «аргументы → байты»;
- Классы констант (UpdateField, InventorySlots, DisplayData, WorldOpcode-сопутствующие);
- Чистая математика/таблицы (TalentMath, SpellCastTimes, SpellDurations, BagInventory-адресация);
- Extension-методы;
- Крипто-математика (Srp6, WorldAuth).

**Запрещено:** статики с мутабельным состоянием (кэши, реестры), с I/O (БД, сеть), с оркестрацией игровой логики. Такое — DI-сервис (см. §5). Состояние per-connection живёт в `WorldSession`, глобальное — в `WorldState`/сторах.

## 5. DI-конвенции

- Контейнер — `Microsoft.Extensions.DependencyInjection`, регистрация в `Program.cs` хоста.
- Сервисы — **stateless синглтоны** (всё состояние — в WorldSession/WorldState): ни одного мутабельного поля в сервисе; зависимость только через конструктор.
- Опкод-хендлеры — **модули**: `internal sealed class XxxHandlers : IOpcodeHandlerModule`, методы с `[WorldOpcodeHandler(opcode)]`; модули регистрируются assembly-сканом (`AddWorldOpcodeHandlers()`), диспетчеризация — `WorldPacketRouter` (DI-синглтон). Добавить опкод = добавить метод с атрибутом.
- Объекты с ручным временем жизни (per-connection `WorldSession`) создаются **фабрикой** (`WorldSessionFactory`) с **parameter object** (`WorldSessionServices`). Service locator (`IServiceProvider` в полях) — запрещён.
- Узкие интерфейсы (ISP): потребитель зависит от `IFactionRepository`, а не от фасада, если ему нужна одна область. Фасады (`IWorldRepository`) — только там, где потребителю реально нужно много областей.
- Паттерны применяем по необходимости (фабрика, фасад, command для dev-команд, реестр для опкодов); не плодим абстракции «на вырост» (KISS).

## 6. Async

- Везде `async`/`await`, без `.Result`/`.Wait()`.
- Fire-and-forget — только осознанно: `_ = Task.Run(...)` с логированием исключений внутри.
- Замыкания в `Task.Run` не должны захватывать ничего со временем жизни короче синглтона, кроме session (живёт дольше задачи).

## 7. Обработка ошибок

- Containment: исключение одного тика/одной сессии не валит сервер — ловим на границе (цикл тика, цикл сессии), логируем с контекстом, продолжаем.
- Исключения не глотаются молча: минимум `LogError` с причиной и идентификаторами (аккаунт, guid, spellId).
- Валидация клиентского ввода — на входе хендлера; сервер авторитетен (reassert состояния клиенту при расхождении).

## 8. Протокол и эталоны

- Структуры пакетов сверяются с эталонами: CMaNGOS-WotLK (основной), TrinityCore 3.3.5. В нетривиальных местах комментарий со ссылкой на эталонный метод (`WorldSession::HandleMoveWorldportAck`).
- Справочник опкодов/структур: gtker.com (wowdev отдаёт 403).
- Константы протокола переносятся дословно, с комментарием-источником; «магические» числа без источника не допускаются.

## 9. Тесты и проверка

- Новая чистая логика (крипто, математика, парсеры) — покрывается xUnit-тестами (`tests/`).
- Игровая логика проверяется сборкой (`dotnet build`, warnings = errors), запуском сервера (число зарегистрированных опкодов в логе старта — инвариант при рефакторингах) и клиентом 3.3.5a.
- Один логический срез работы = один коммит (`fix(M11): …` / `refactor(M7): …`), журнал граблей — `docs/archive/devlog/`.
