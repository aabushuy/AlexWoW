## БАГ: экипировка чужих игроков не отрисовывается у наблюдателя (интермиттентно)

**Статус:** 🔴 открыт (отложен из M6.1-сессии).
**Симптом:** игрок A видит игрока B **голым** (нижнее бельё), хотя у B надет стартовый набор.
Своя экипировка у каждого отображается всегда. Воспроизводится с двумя клиентами рядом.
Интермиттентно: иногда после релога наблюдателя одежда соседа появляется, чаще — нет.
Закономерность по наблюдениям пользователя: чаще «голым» виден тот, кто залогинился раньше
(его create приходит наблюдателю в момент загрузки), но проявление нестабильное.

**Что установлено диагностикой (логи `[vis]`, `[item-query]`):**
- `SMSG_UPDATE_OBJECT` (CREATE_OBJECT2) для соседа уходит наблюдателю **с полной экипировкой**
  (лог `CREATE 'Magic'(eq=5) → 'Humantest'`), и не один раз (вход + refresh на движении).
- Клиент наблюдателя **не шлёт `CMSG_ITEM_QUERY_SINGLE`** на предметы соседа — значит модель
  он должен строить из локального Item.dbc по `PLAYER_VISIBLE_ITEM_*_ENTRYID`, без запроса.
- Объект соседа создаётся (юнит виден, имя/класс/раса верные) — теряются только видимые предметы.

**Что НЕ помогло:**
- `socket.NoDelay=true` (сгладило движение, но не одежду).
- Динамическая видимость игроков (пере-создание на движении) — create повторяется, но клиент
  **игнорирует повторный CREATE для уже существующего объекта**.
- Отложенная (1.5с/4с после входа) досылка **VALUES-апдейта** только с полями
  `PLAYER_VISIBLE_ITEM_*_ENTRYID` (`PlayerSpawn.BuildEquipmentValuesUpdate`) — тоже не применилась.

**Рабочая гипотеза:** гонка на КЛИЕНТЕ — видимые предметы приходят раньше, чем достроена модель
соседа, и игнорируются; последующие апдейты не «переодевают». Но VALUES-досылка должна была
сработать — значит либо формат VALUES-апдейта/индексы неверны, либо нужен иной механизм.

**Что проверить завтра (план атаки):**
1. **Сверить байты** нашего CREATE для чужого игрока с эталоном (TrinityCore/CMaNGOS
   `BuildCreateUpdateBlockForPlayer` для другого игрока): порядок полей, `PLAYER_VISIBLE_ITEM_*`
   (entry + **enchant**-поле), нужен ли `UNIT_VIRTUAL_ITEM_SLOT_*` / иные поля для отрисовки.
2. **Проверить формат VALUES-апдейта** (UPDATETYPE_VALUES=0): возможно, для игрока нужен иной
   набор/маска, либо клиент применяет экипировку только из CREATE-блока. Снять дамп пакета.
3. **Исключить клиентский кэш**: тест с ДВУХ разных машин/инсталляций (сейчас, возможно, два
   клиента на одной машине делят `Cache/WDB` → артефакт).
4. Проверить, не нужен ли клиенту полный набор полей юнита (например, `UNIT_FIELD_BYTES_2`,
   `PLAYER_BYTES` у чужого) для сборки экипированной модели.
5. Возможно, корректный путь — отрисовка экипировки придёт «бесплатно», когда видимость игроков
   переедет в грид+тик (M6.8/M6.3) и состояние будет досылаться стабильно после загрузки.

**Связанные наработки в коде (оставлены, могут пригодиться):**
- `WorldState.RefreshVisiblePlayersAsync` (динамическая видимость игроков — работает: соседи
  появляются/исчезают при перемещении).
- `WorldState.ResendNearbyEquipmentToAsync` + отложенный вызов в `WorldEntryHandlers.OnPlayerLogin`.
- `PlayerSpawn.BuildEquipmentValuesUpdate`.
- Диагностические логи `[vis]`/`[item-query]` (понижены до Debug).

## Чистота: лишние тест/кастом-NPC у старта (вендоры, арена-мастера и пр.)

**Статус:** 🟡 открыт.
**Симптом:** у стартовой зоны (Долина Северозёмья) кучей стоят кастом/тест-NPC дампа CMaNGOS,
которых не должно быть в ванильном старте:
- generic-вендоры **«Weapons Vendor»** без личного имени;
- **«Arena Battlemaster» / Arena Organizer / Arena Vendor** и пр. арена-NPC (кастом CMaNGOS,
  заспавнены «в каждом городе»; subname `Arena Battlemaster` и т.п.).
Аналогично ранее вычищенным generic-тренерам/`[DND]` и точечно исключённым арена-entry (M5.6b,
`WorldDatabase.ExcludedCreatureEntries` = 26012/26075/26760).
**Фикс:** расширить фильтр спавна в `WorldDatabase.GetCreaturesNearAsync`:
- generic-вендоры без subname (`Name LIKE '% Vendor' AND COALESCE(SubName,'')=''`);
- арена-NPC по subname (`SubName LIKE 'Arena %'` → Battlemaster/Organizer/Vendor, + Brutal/Test Arena);
- при необходимости дополнить `ExcludedCreatureEntries` точечными id.
Проверить, что настоящие именованные торговцы/мастера остаются.
