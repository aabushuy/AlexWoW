## БАГ: экипировка чужих игроков не отрисовывается у наблюдателя (интермиттентно)

**Статус:** 🔴 открыт (отложен из M6.1-сессии).
**Симптом:** игрок A видит игрока B **голым** (нижнее бельё), хотя у B надет стартовый набор.
Своя экипировка у каждого отображается всегда. Воспроизводится с двумя клиентами рядом.
Интермиттентно: иногда после релога наблюдателя одежда соседа появляется, чаще — нет.
Закономерность по наблюдениям пользователя: чаще «голым» виден тот, кто залогинился раньше
(его create приходит наблюдателю в момент загрузки), но проявление нестабильное.

**Что установлено диагностикой (логи `[vis]`, `[item-query]`):**
- `SMSG_UPDATE_OBJECT` (CREATE_OBJECT2) для соседа уходит наблюдателю **с полной экипировкой**
  (лог `CREATE 'Magic'(eq=5) → 'Humantest'`), и не один раз (вход + refresh на движении).
- Клиент наблюдателя **не шлёт `CMSG_ITEM_QUERY_SINGLE`** на предметы соседа — значит модель
  он должен строить из локального Item.dbc по `PLAYER_VISIBLE_ITEM_*_ENTRYID`, без запроса.
- Объект соседа создаётся (юнит виден, имя/класс/раса верные) — теряются только видимые предметы.

**Что НЕ помогло:**
- `socket.NoDelay=true` (сгладило движение, но не одежду).
- Динамическая видимость игроков (пере-создание на движении) — create повторяется, но клиент
  **игнорирует повторный CREATE для уже существующего объекта**.
- Отложенная (1.5с/4с после входа) досылка **VALUES-апдейта** только с полями
  `PLAYER_VISIBLE_ITEM_*_ENTRYID` (`PlayerSpawn.BuildEquipmentValuesUpdate`) — тоже не применилась.

**Рабочая гипотеза:** гонка на КЛИЕНТЕ — видимые предметы приходят раньше, чем достроена модель
соседа, и игнорируются; последующие апдейты не «переодевают». Но VALUES-досылка должна была
сработать — значит либо формат VALUES-апдейта/индексы неверны, либо нужен иной механизм.

**Что проверить завтра (план атаки):**
1. **Сверить байты** нашего CREATE для чужого игрока с эталоном (TrinityCore/CMaNGOS
   `BuildCreateUpdateBlockForPlayer` для другого игрока): порядок полей, `PLAYER_VISIBLE_ITEM_*`
   (entry + **enchant**-поле), нужен ли `UNIT_VIRTUAL_ITEM_SLOT_*` / иные поля для отрисовки.
2. **Проверить формат VALUES-апдейта** (UPDATETYPE_VALUES=0): возможно, для игрока нужен иной
   набор/маска, либо клиент применяет экипировку только из CREATE-блока. Снять дамп пакета.
3. **Исключить клиентский кэш**: тест с ДВУХ разных машин/инсталляций (сейчас, возможно, два
   клиента на одной машине делят `Cache/WDB` → артефакт).
4. Проверить, не нужен ли клиенту полный набор полей юнита (например, `UNIT_FIELD_BYTES_2`,
   `PLAYER_BYTES` у чужого) для сборки экипированной модели.
5. Возможно, корректный путь — отрисовка экипировки придёт «бесплатно», когда видимость игроков
   переедет в грид+тик (M6.8/M6.3) и состояние будет досылаться стабильно после загрузки.

**Связанные наработки в коде (оставлены, могут пригодиться):**
- `WorldState.RefreshVisiblePlayersAsync` (динамическая видимость игроков — работает: соседи
  появляются/исчезают при перемещении).
- `WorldState.ResendNearbyEquipmentToAsync` + отложенный вызов в `WorldEntryHandlers.OnPlayerLogin`.
- `PlayerSpawn.BuildEquipmentValuesUpdate`.
- Диагностические логи `[vis]`/`[item-query]` (понижены до Debug).

## Чистота: лишние тест/кастом-NPC у старта (вендоры, арена-мастера и пр.)

**Статус:** 🟡 открыт.
**Симптом:** у стартовой зоны (Долина Северозёмья) кучей стоят кастом/тест-NPC дампа CMaNGOS,
которых не должно быть в ванильном старте:
- generic-вендоры **«Weapons Vendor»** без личного имени;
- **«Arena Battlemaster» / Arena Organizer / Arena Vendor** и пр. арена-NPC (кастом CMaNGOS,
  заспавнены «в каждом городе»; subname `Arena Battlemaster` и т.п.).
Аналогично ранее вычищенным generic-тренерам/`[DND]` и точечно исключённым арена-entry (M5.6b,
`WorldDatabase.ExcludedCreatureEntries` = 26012/26075/26760).
**Фикс:** расширить фильтр спавна в `WorldDatabase.GetCreaturesNearAsync` (и аналогично для
гейм-объектов в `GetGameObjectsNearAsync`):
- generic-вендоры без subname (`Name LIKE '% Vendor' AND COALESCE(SubName,'')=''`);
- арена-NPC по subname (`SubName LIKE 'Arena %'` → Battlemaster/Organizer/Vendor, + Brutal/Test Arena);
- **`[DND]` в любом месте имени** (сейчас фильтр ловит только ведущий `[`, а есть, напр.,
  «Lottery Commissioner [DND]» и [DND] в середине; ~8 таких гейм-объектов): заменить
  `Name NOT LIKE '[%'` на также `Name NOT LIKE '%[DND]%'` (плюс `%[PH]%`/`%[UNUSED]%`);
- при необходимости дополнить `ExcludedCreatureEntries` точечными id.
Проверить, что настоящие именованные торговцы/мастера остаются.

## БАГ: ruRU-клиент спрашивает склонения имени каждый вход

**Статус:** 🔴 открыт (первый фикс не помог).
**Симптом:** на экране выбора персонажа ruRU-клиент каждый раз показывает диалог «Склонение имени»
для кириллических имён, хотя игрок уже их вводил.

**Что сделано (не помогло):**
- `CMSG_SET_PLAYER_DECLINED_NAMES (0x419)` парсится и **сохраняется** (таблица
  `character_declined_names` — проверено, запись есть).
- `SMSG_NAME_QUERY_RESPONSE` теперь возвращает `has_declined_names=1` + 5 склонений, если заданы.

**Почему не сработало (диагностика):** на экране выбора клиент **НЕ шлёт `CMSG_NAME_QUERY (0x50)`**
для своего персонажа (в логах его нет) — значит ответ со склонениями там не запрашивается, и триггер
диалога иной.

**Гипотезы / план на потом:**
1. Триггер — в `SMSG_CHAR_ENUM`: проверить, нет ли в layout per-char поля/флага про declined names
   (сверить с wow_messages `smsg_char_enum` 3.3.5 и как CMaNGOS заполняет). Возможно, надо слать
   declined-блок прямо в enum.
2. Снять опкоды, которые клиент шлёт ИМЕННО при показе диалога/на char-select (вкл. 0x182/0x11C/0x1F4
   без обработчиков) — возможно, один из них и есть запрос склонений/состояния.
3. Проверить, что `CMSG_NAME_QUERY` (если шлётся в мире) читает guid правильно (packed vs u64).
4. Возможно, клиент кеширует склонения локально только после name-query — тогда нужно, чтобы клиент
   их запрашивал (или слать proactively).

- **Ручные индексы `mangos`**: `idx_map_x (map, position_x)` на `creature`/`gameobject` добавлены
  вручную; при пересоздании тома `alexwow-mysql-data` и переимпорте дампа — создать заново.
  Перспектива — кэш видимости в памяти (см. [M6.8](M6.8-perf-spatial-cache.md)).
- **Движение соседей «прыгает»** (рассинхрон часов) — план фикса в [M6.3](M6.3-combat.md)
  (нормализация времени через time-sync на серверном тике).

## БАГ: повторный ввод пароля вешает окно авторизации

**Статус:** 🔴 открыт.
**Симптом:** на экране логина ввести НЕВЕРНЫЙ пароль, затем ВЕРНЫЙ — окно «Авторизация» виснет,
нужен перезапуск клиента. Вторая попытка SRP6 после отказа не обрабатывается.
**Где копать:** `AlexWoW.AuthServer` (realmd/SRP6): после кода ошибки на LOGON_PROOF сервер,
вероятно, не сбрасывает состояние / не принимает повторный `CMD_AUTH_LOGON_CHALLENGE`/`_PROOF`
на том же TCP. Проверить переиспользование сессии после неуспеха и повторный challenge на коннекте.

## БАГ (M6.9): не снимается надетая экипировка

**Статус:** 🟡 частично (надевание/сплит/перемещение/выброс — ОК; снятие — нет).
**Диагностика:** при попытке снять вещь сервер **не получает** инвентарь-пакет (в логах нет
`[inv] SWAP_INV/SWAP_ITEM/AUTOSTORE` с src в слоте экипировки 0..18; необработанных опкодов нет;
рюкзак не полон). Клиент либо блокирует локально, либо шлёт `AUTOEQUIP` с уже надетого слота.
**Сделано:** `OnAutoEquipItem` снимает, если src — слот экипировки. Не помогло.

**ПОДТВЕРЖДЕНО (логи живого коннекта):** `сумка→экипировка` шлёт `SWAP_INV src=<bag> dst=<equip>`
и работает; `экипировка→сумка` (drag или right-click по надетой вещи) **не шлёт НИЧЕГО** на сервер.

**Доп. наблюдения (2026-06-04):**
- В слоте экипировки **видна иконка предмета И тултип** при наведении → item-object и
  `PLAYER_FIELD_INV_SLOT` для слота экипировки у клиента **валидны** (не пустой слот).
- **Замена работает:** клик по вещи такого же типа в сумке (напр. другая «грудь») → надевается
  новая, старая уходит в сумку (это AUTOEQUIP src=<bag>, dst занят → своп; сервер кладёт старую в сумку — ОК).
- Ломается ТОЛЬКО **прямое снятие** (equip-слот как источник drag / right-click) — клиент не
  инициирует пакет. Т.е. дело не в данных слота/предмета, а в том, что клиент отказывается делать
  слот экипировки источником перемещения «в сумку».
- Гипотезы сузились: возможно недостающий флаг/поле предмета (`ITEM_FIELD_FLAGS`, durability) либо
  состояние игрока; либо клиент ждёт иной механизм. Снять дамп исходящего пакета при right-click
  по надетой вещи (если он вообще есть) и сверить с эталоном.

**План атаки (нужен protocol-level разбор):**
1. Сверить с CMaNGOS/TrinityCore, что именно делает экипированный предмет «кликабельным»: вероятно,
   нужен полноценный item-object у клиента с корректными `ITEM_FIELD_OWNER`/`CONTAINED` И, возможно,
   пере-создание item-object при перемещении в слот (а не только VALUES-апдейт `PLAYER_FIELD_INV_SLOT`).
2. Проверить, появляется ли иконка предмета В слоте paperdoll (а не только на 3D-модели) — если нет,
   значит `PLAYER_FIELD_INV_SLOT` для слотов экипировки у клиента не применяется в рантайме.
3. Снять дамп исходящих пакетов клиента при попытке снять (Wireshark/клиентский лог), сверить опкод.
4. Возможно, дело в недостающих полях юнита/игрока, нужных для интерактивности экипировки.
