# M6.8 — Производительность: кэш видимости (грид в памяти; Redis при масштабе)

**Статус:** ⬜ запланировано (рефакторинг, не геймплей).
**Зависит от:** ничего; пересекается с M6.3 (серверный тик) и видимостью M5.

## Проблема

Видимость NPC/гейм-объектов пересчитывается **запросом в БД мира на каждый ~20-ярдовый шаг**
игрока (`SpawnHandlers.RefreshVisibleNpcsAsync`/`...GameObjectsAsync` в `MovementHandlers`).
Стопгап-меры уже сделаны (см. ниже), но per-move запросы в БД **не масштабируются**: с ростом
онлайна десятки игроков начнут долбить спатиал-запросами одну БД → задержки/рывки вернутся.

### Уже сделано (стопгап, M6.1-сессия)

- `socket.NoDelay = true` (отключён Nagle) — основная причина рывков/«полёта» соседей.
- Составной индекс `(map, position_x)` на `mangos.creature` и `mangos.gameobject` — ускоряет
  спатиал-выборку. ⚠️ Индексы добавлены **вручную**; при пересоздании тома `alexwow-mysql-data`
  и переимпорте дампа их нужно **создать заново** (как и сам дамп).

## Целевое решение

1. **Спатиал-грид в памяти процесса (primary).** Грузить спавны (creature/gameobject) карты в
   память **один раз** (при старте или лениво по карте) в грид-ячейки (напр. 100×100 ярдов).
   Видимость = выборка соседних ячеек из памяти, **без БД в горячем пути**. Это убирает per-move
   запросы целиком и на порядок дешевле. Сюда же логично переехать пересчёту видимости из
   обработчика движения в **серверный тик** (появится в M6.3).
2. **Redis — при горизонтальном масштабе.** Когда world-серверов станет несколько (шардинг карт
   или несколько процессов), общее состояние (живые сущности, позиции игроков, агро) выносится в
   общий стор. Redis (или аналог) — кандидат: pub/sub для кросс-нодовой видимости/рассылки,
   общие счётчики GUID. Для **одного** процесса Redis избыточен — in-process грид быстрее (нет
   сетевого хопа).

## Задачи (когда дойдём)

- Загрузчик спавнов в память (по карте), грид-индекс ячеек, инвалидация на смерть/респаун (M6.3).
- Перевод `RefreshVisible*` на грид; вынос пересчёта в мировой тик.
- (Масштаб) абстракция «реестр мира» с возможной Redis-реализацией; шардинг по картам.

## Критерий

Бег игрока не порождает запросов в БД мира; видимость плавная при N одновременных игроков
(нагрузочно). Поведение клиента не меняется (тот же CREATE/DESTROY-диф).
