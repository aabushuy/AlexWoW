# M6.1 — Стартовая экипировка и инвентарь

**Статус:** 🟡 в работе
**Зависит от:** M5 (вход в мир, спавн игрока, UpdateMask/PackedGuid).
**Эталон:** CMaNGOS-WotLK (`Player::_LoadInventory`, `Player::FindEquipSlot`,
`Item`, `playercreateinfo_item`, `item_template`).

## Цель / критерий приёмки (клиент 3.3.5a)

Сейчас персонаж входит в мир **голым**. Нужно: новый персонаж появляется **одетым**
(шмот/оружие видны на 3D-модели и у других игроков), предметы лежат в слотах экипировки
и рюкзаке окна персонажа, тултипы предметов корректны (имя/иконка/тип). Предметы
**персистентны** (сохраняются между входами).

Это фундамент под M6.2 (торговля — продавать/покупать предметы) и далее.

## Данные

### БД мира `mangos` (read-only, `WorldDatabase`)

- **`item_template`** — характеристики предмета: `entry, class, subclass, name, displayid,
  Quality, Flags, BuyPrice, SellPrice, InventoryType, ItemLevel, RequiredLevel, MaxDurability,
  stackable, ...` (для item-query и маппинга в слот экипировки).
- **`playercreateinfo_item`** (`race, class, itemid, amount`) — стартовый набор по расе/классу.
  ⚠️ В этом дампе CMaNGOS таблица **пуста**: базовый стартовый outfit в CMaNGOS берётся из
  клиентского **`CharStartOutfit.dbc`** (по race/class/gender → до 24 itemId), а
  `playercreateinfo_item` — лишь доп. предметы. Поэтому набор извлекаем из DBC офлайн (как
  maps/vmaps) и заливаем в `mangos.playercreateinfo_item` (см. ниже), сервер читает из БД.

Новые методы:
- `GetStartingItemsAsync(byte race, byte class)` → список (itemEntry, amount).
- `GetItemTemplateAsync(uint entry)` → `ItemTemplateData?`.

### Извлечение стартового набора из `CharStartOutfit.dbc` (офлайн, `tools/MapExtractor`)

Новая команда `charstartoutfit <dataDir> <out.sql>`: читает `DBFilesClient\CharStartOutfit.dbc`
(WDBC; запись: id u32, packed u32 [byte0=race, byte1=class, byte2=gender], itemId[24] i32,
displayId[24] i32, invType[24] i32) → дедуп по (race,class,itemid) → SQL
`INSERT INTO playercreateinfo_item (race,class,itemid,amount) VALUES …;`. Залить в `mangos` на
homeserver (одноразово, как импорт дампа мира). Клиентские данные в репо не кладём.

### БД `characters` (`CharactersDatabase`)

Новая таблица (в `EnsureSchemaAsync`):
```sql
CREATE TABLE IF NOT EXISTS character_items (
  item_guid   INT UNSIGNED NOT NULL AUTO_INCREMENT,
  owner_guid  INT UNSIGNED NOT NULL,
  item_entry  INT UNSIGNED NOT NULL,
  bag         TINYINT UNSIGNED NOT NULL DEFAULT 255,  -- 255 = INVENTORY_SLOT_BAG_0 (основной)
  slot        TINYINT UNSIGNED NOT NULL,              -- экипировка 0..18, рюкзак 23..38
  stack_count INT UNSIGNED NOT NULL DEFAULT 1,
  PRIMARY KEY (item_guid),
  KEY ix_items_owner (owner_guid)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
GUID предмета = `item_guid (low)` + `HIGHGUID_ITEM` (константа 3.3.5a — сверить с CMaNGOS).

CRUD: `GetItemsAsync(ownerGuid)`, `AddItemAsync(ownerGuid, entry, bag, slot, count)`,
`RemoveItemAsync(itemGuid)`, `MoveItemAsync(itemGuid, bag, slot)` (последние два — задел под M6.2).

## Слоты (3.3.5a)

Константы (`InventorySlots.cs`): экипировка `EQUIPMENT_SLOT_START=0 .. EQUIPMENT_SLOT_END=19`,
сумки `INVENTORY_SLOT_BAG_START=19 .. END=23`, рюкзак `INVENTORY_SLOT_ITEM_START=23 .. END=39`.

Маппинг `item_template.InventoryType` → equipment slot (как `Player::FindEquipSlot`):
HEAD=0, NECK=1, SHOULDERS=2, BODY(рубаха)=3, CHEST/ROBE=4, WAIST=5, LEGS=6, FEET=7, WRISTS=8,
HANDS=9, FINGER=10/11, TRINKET=12/13, BACK(плащ)=14, MAINHAND/WEAPON/2H=15, OFFHAND/SHIELD/HOLDABLE=16,
RANGED/THROWN/RELIC=17, TABARD=18. Неэкипируемое (еда/вода/реагенты) → рюкзак (23+).

## Протокол

### Опкоды (`WorldOpcode.cs`)
- `CmsgItemQuerySingle = 0x056`
- `SmsgItemQuerySingleResponse = 0x058`

### UpdateField (`UpdateField.cs`) — индексы из CMaNGOS `UpdateFields.h` (3.3.5a)
- **ITEM** (после `OBJECT_END=0x06`): `ITEM_FIELD_OWNER`, `ITEM_FIELD_CONTAINED`,
  `ITEM_FIELD_STACK_COUNT`, `ITEM_FIELD_DURABILITY`, `ITEM_FIELD_MAXDURABILITY`.
- **PLAYER**: `PLAYER_FIELD_INV_SLOT_HEAD` (начало слотов: экипировка+сумки+рюкзак, по 2 поля = GUID),
  `PLAYER_VISIBLE_ITEM_1_ENTRYID` (19 видимых, stride 2: entry + enchant) — **одевают модель**,
  `PLAYER_FIELD_COINAGE` (деньги — задел под M6.2).

### `ItemObject.BuildCreateObject(item)` (новый `Protocol/ItemObject.cs`)
`SMSG_UPDATE_OBJECT`, `UPDATETYPE_CREATE_OBJECT2`, `TYPEID_ITEM (1)`, **без** movement-блока
(item — не Living, флаг update = 0). Values: GUID, `OBJECT_FIELD_ENTRY`, type-mask `Object|Item (0x3)`,
scale=1, `ITEM_FIELD_OWNER`/`ITEM_FIELD_CONTAINED` = player guid, `ITEM_FIELD_STACK_COUNT`,
durability/maxdurability.

### `PlayerSpawn.BuildValues`
- Self: `PLAYER_FIELD_INV_SLOT_HEAD + slot*2` = GUID предмета (занятые слоты), плюс
  `PLAYER_VISIBLE_ITEM_n_ENTRYID` = entry для экипированных (видимость шмота).
- Чужой игрок (спавн соседям): только видимые предметы (visible item), guid-слоты не нужны.

### `CharEnum.BuildBody`
Для каждого из 23 слотов писать `displayInfoId(4) + invType(1) + enchant(4)` из экипировки
(paperdoll на экране выбора). Нужны `item_template.displayid` + `InventoryType` экипированных.

### `CMSG_ITEM_QUERY_SINGLE → SMSG_ITEM_QUERY_SINGLE_RESPONSE` (новый `Handlers/ItemHandlers.cs`)
Ответ из `item_template` (layout сверить с gtker `smsg_item_query_single_response`):
entry, class, subclass, sound_override, name[4], displayid, Quality, Flags, Flags2, BuyPrice,
SellPrice, InventoryType, ... «Нет данных» → `entry | 0x80000000`.

## Поток

- **Создание персонажа** (`CharScreenHandlers.TryCreateAsync`): после `CreateAsync` — выдать
  стартовый набор (`playercreateinfo_item` → разложить по слотам → `character_items`).
- **Вход голым** (`WorldEntryHandlers.OnPlayerLogin`): если `character_items` пусто — выдать набор
  (миграция текущих тестовых персонажей). Загрузить инвентарь в сессию.
- **Спавн при входе**: создать item-объекты (`ItemObject`) у клиента, затем self `PlayerSpawn`
  с заполненными слотами/видимыми предметами. Порядок проверить на клиенте.
- **`WorldSession`**: кэш инвентаря сессии (для item-query без БД и под M6.2).

## Проверка

1. `dotnet build` чисто.
2. Деплой; БД `mangos` уже содержит `item_template`/`playercreateinfo_item`.
3. Живой клиент (TEST/test): новый персонаж — одет на экране выбора и в мире; окно C — слоты
   экипировки заполнены; рюкзак B — еда/вода; тултипы корректны; релог — предметы на месте;
   второй клиент видит одетую модель соседа.

## Грабли / на что смотреть
- Видимость шмота на модели даёт `PLAYER_VISIBLE_ITEM_n_ENTRYID`, **не** слот-GUID.
- Клиент может требовать item-объекты до ссылки на их GUID в слотах — при проблемах слать
  item-объекты раньше self-update либо одним пакетом с несколькими блоками.
- `SMSG_ITEM_QUERY_SINGLE_RESPONSE` кэшируется клиентом по entry.
- HIGHGUID_ITEM и точные индексы UpdateFields — сверить с CMaNGOS, не угадывать.
