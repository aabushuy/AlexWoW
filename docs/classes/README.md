# Классы — трекинг абилок и талантов (WoW 3.3.5a)

Здесь по каждому классу два файла:
- **`<класс>-abilities.md`** — абилки/заклинания из книги заклинаний: вкладка **«Общее»** + 3 вкладки
  (школы/специализации). Что проверено клиентом, что осталось.
- **`<класс>-talents.md`** — таланты по 3 деревьям: активные (дают абилку/меняют механику) и ключевые
  пассивные (влияют на расчёты — для нас это `SpellModifier`-ауры 107/108 и спец-эффекты).

## Легенда статуса

| Знак | Значение |
|---|---|
| ✅ | реализовано и **проверено клиентом** |
| 🟡 | реализовано (в каталоге/коде), проверки клиентом нет |
| ⬜ | не сделано / не проверено |
| ➖ | не планируется на этом этапе (петы, тотемы-как-сущности, формы-модели и т.п.) |

## Колонки

- **Спелл** — rank 1 spell_id (высшие ранги — те же поля spell_template, гейтятся уровнем; см. M10.3).
- **Школа/Аура** — школа магии (Physical/Fire/Frost/Arcane/Nature/Shadow/Holy) **или** тип ауры
  (`EffectApplyAuraName`, CMaNGOS `SpellAuraDefines.h`) — чтобы при правке сразу видеть механику.
- **Тип** — Direct (прямой урон/хил), DoT/HoT (периодика), Buff/Debuff (аура), Toggle (стойка/аура/аспект,
  группа эксклюзивности), Melee (мили-абилка по оружию), Utility (телепорт/призыв/прочее).

## Как мы это используем

Движок спеллов — data-driven из `spell_template` (M10.2): урон/школа/стоимость/кулдаун/эффекты берутся из
дампа, хардкода нет. Поэтому «реализовать абилку» часто = убедиться, что её эффект-тип распознан в
`SpellCatalog.FromTemplate` и применяется (`SpellEffectsService`/`PeriodicsService`/`AuraService`).
Незнакомые типы аур/эффектов — кандидаты на доработку (как `MOD_BLOCK_PERCENT`, `MOD_DAMAGE_PERCENT_TAKEN`).
