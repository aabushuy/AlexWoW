# AlexWoW — Аналитика проекта

> Свой сервер World of Warcraft на **.NET** + **MySQL**.
> Цель: **играбельный сервер** под **WotLK 3.3.5a (build 12340)**.
> Эталон для портирования: **CMaNGOS-WotLK** (`cmangos/mangos-wotlk`), дополнительно TrinityCore 3.3.5 и WCell (C#).

---

## 1. Что такое эмулятор сервера WoW

Эмулятор — это набор сетевых служб, которые «притворяются» официальным сервером для немодифицированного клиента `Wow.exe` (3.3.5a, build 12340). Клиент сам по себе содержит всю графику/звук/карты; сервер отвечает за аутентификацию, состояние мира и игровую логику.

```
┌──────────────┐   1. SRP6 login    ┌───────────────┐
│   Wow.exe    │ ◄────────────────► │  AuthServer   │  логин + список реалмов
│  build 12340 │      (TCP 3724)    │  (realmd)     │
│              │                    ├───────────────┤
│              │   2. world proto   │  WorldServer  │  вся игровая логика
│              │ ◄────────────────► │  (mangosd)    │  (TCP 8085)
└──────────────┘   (RC4 encrypted)  └───────┬───────┘
                                            │
                          ┌─────────────────┼─────────────────┐
                      MySQL             DBC / maps         (вынесено из
                  auth/chars/world   (данные клиента)        клиента)
```

## 3. Сетевой протокол 3.3.5a (ключевые узлы)

### 3.1 Аутентификация — SRP6
- Логин-сервер использует **SRP6** (Secure Remote Password): обмен `B`, `N`, `g`, `s`; клиент доказывает знание пароля без его передачи.
- Результат — **session key (K)**, который потом используется для шифрования world-трафика.
- Логины/пароли хранятся как SRP-верификатор `v` и соль `s` (не как хэш пароля).

### 3.2 Шифрование world-пакетов — RC4 (важное отличие от ванили!)
- В **3.3.5 заголовки пакетов шифруются RC4**, ключи выводятся через **HMAC-SHA1** из session key.
  - Серверный ключ (encrypt): HMAC-SHA1(K) с серверным seed.
  - Клиентский ключ (decrypt): HMAC-SHA1(K) с клиентским seed.
  - Делается «прогрев» (drop) первых ~1024 байт RC4-потока.
- Шифруется только **заголовок** (размер + opcode), тело пакета — открытое.
- Без корректной реализации этого шага клиент не войдёт в мир (разорвёт соединение).

### 3.3 Opcodes
- Сотни опкодов: `CMSG_*` (от клиента), `SMSG_*` (к клиенту), `MSG_*` (двусторонние).
- Архитектура обработки: **диспетчер opcode → handler**. В .NET удобно делать через атрибуты на методах (как в WCell) + реестр.
- Старт-набор для «входа в мир»: `CMSG_AUTH_SESSION`, `SMSG_AUTH_RESPONSE`, `CMSG_CHAR_ENUM`/`SMSG_CHAR_ENUM`, `CMSG_PLAYER_LOGIN`, `SMSG_LOGIN_VERIFY_WORLD`, набор начальных `SMSG_*` (account data, tutorial, time).

### 3.4 Сериализация
- Бинарный little-endian; есть **bit-packed** структуры (особенно movement в 3.3.5).
- **ObjectUpdate (SMSG_UPDATE_OBJECT)** — самый сложный пакет: update-mask поля объектов, движение, создание/обновление сущностей. Это сердце «видимости» мира.

## 4. Данные

### 4.1 Клиентские данные (извлекаются из клиента 3.3.5a)
- **DBC** (`DBFilesClient/*.dbc`) — спеллы, предметы, карты, расы/классы и т.д. Нужно парсить.
- **maps / vmaps / mmaps** — геометрия для коллизий и навигации (pathfinding). Извлекаются утилитами-экстракторами из MPK/MPQ клиента.
- Эти данные **не распространяем** — пользователь генерирует их из своей копии клиента.

### 4.2 Серверная БД мира
- В CMaNGOS-WotLK мир поставляется **MySQL-дампом** (спавны, лут, квесты, тексты, скрипты) — грузим почти как есть, **конвертация не нужна** (ключевая причина выбора MySQL).
- Три логические БД: `auth`/`realmd` (аккаунты, реалмы), `characters` (персонажи, инвентарь), `world`/`mangos` (статические данные мира).
- Свои таблицы (наши служебные) кладём через миграции поверх импортированных дампов мира.

## 5. Технологический стек (.NET)

| Слой | Предлагаемый выбор | Обоснование |
|---|---|---|
| Рантайм | **.NET 8/9** | Современный async, высокая производительность |
| Сеть | **System.IO.Pipelines** + Sockets | Эффективный парсинг бинарного потока без лишних аллокаций |
| Криптография | `System.Security.Cryptography` | SHA1, HMAC-SHA1, RC4 (RC4 реализуем вручную — нет в BCL) |
| Большие числа (SRP6) | `System.Numerics.BigInteger` | Модульная арифметика SRP |
| Доступ к БД | **Dapper** (горячие пути) + **EF Core** (CRUD/миграции) | Скорость + удобство |
| Драйвер MySQL | **MySqlConnector** (+ Pomelo EF Core provider) | Быстрый async-драйвер; совместим с дампами эталонов |
| DI / хостинг | `Microsoft.Extensions.Hosting` | BackgroundService под auth/world |
| Логи | Serilog | Структурные логи |

### Предлагаемая структура решения
```
AlexWoW.sln
├─ src/
│  ├─ AlexWoW.Common        // утилиты, конфиг, логирование
│  ├─ AlexWoW.Cryptography  // SRP6, RC4 header crypt, хэши
│  ├─ AlexWoW.Network       // сокеты, Pipelines, пакет-ридер/райтер
│  ├─ AlexWoW.Protocol      // опкоды, структуры пакетов, ObjectUpdate
│  ├─ AlexWoW.Database      // MySqlConnector/EF/Dapper, репозитории, миграции
│  ├─ AlexWoW.DataStores    // парсеры DBC, загрузка maps/vmaps/mmaps
│  ├─ AlexWoW.Game          // игровая логика: сущности, грид, бой, спеллы
│  ├─ AlexWoW.AuthServer    // exe: realmd
│  └─ AlexWoW.WorldServer   // exe: mangosd
├─ tools/
│  └─ MapExtractor          // извлечение DBC/maps/vmaps/mmaps из клиента
└─ tests/
```

## 7. Дорожная карта (вехи)

> Каждая веха — проверяемый результат на реальном клиенте 3.3.5a.

- **M0. Каркас** — solution, CI, конфиг, логирование, подключение к MySQL.
- **M1. AuthServer** — SRP6, клиент доходит до экрана выбора реалма. *(изолировано, хороший первый успех)*
- **M2. World handshake** — `CMSG_AUTH_SESSION`, RC4-шифрование, вход в сессию.
- **M3. Экран персонажей** — `CHAR_ENUM`, создание/удаление персонажа (запись в Postgres).
- **M4. Вход в мир** — `PLAYER_LOGIN`, спавн, базовый `ObjectUpdate`, движение, чат.
- **M5. Видимость и мир** — грид/cell, NPC, гейм-объекты, видимость игроков.
- **M6. Игровые системы** — бой, спеллы, квесты, лут, инвентарь… *(итеративно, долго)*

Вехи M0–M4 — реалистичный «играбельный минимум: зайти в мир и побегать». M5–M6 — путь к полноценному играбельному серверу (объём, сопоставимый с многолетней работой сообществ).

## 8. Риски и заметки

- **Объём.** Полноценный играбельный сервер — это годы. Поэтому строго идём по вехам, каждая даёт проверяемый результат.
- **БД — MySQL.** Совпадает с форматом дампов эталонов → конвертация не нужна. Минус: MySQL-специфика (типы, движок InnoDB) — не уходим от неё, закладываемся на это в схеме.
- **Bit-packed movement в 3.3.5.** Один из самых нетривиальных участков протокола — закладываем время.
- **RC4 header crypt.** Критичен для M2; реализуем и тестируем отдельно по эталону.
- **Данные клиента.** DBC/maps/vmaps/mmaps генерируются пользователем из своей копии клиента; в репозиторий не кладём.
- **Правовой статус.** Эмуляция в образовательных целях; данные Blizzard не распространяем.
